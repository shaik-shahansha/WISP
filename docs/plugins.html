Plugins & ROS2

Dynamically extend any device with capabilities at runtime. WISP ships with ROS2Plugin — a live ROS2 graph scanner that turns your robot into a Telegram-controlled agent.

Plugin API

Every plugin subclasses WispPlugin and implements a single method — attach(device). This method is called once when you call device.use(plugin), before the transport starts.

from wisp import WispPlugin, WispDevice, CapabilitySpec

class WeatherPlugin(WispPlugin):
    def attach(self, device: WispDevice) -> None:
        device.registry.register(CapabilitySpec(
            name="get_forecast",
            description="Get 3-day weather forecast for a city.",
            params={"city": {"type": "string"}},
            method=self._fetch,
        ))

    def _fetch(self, city: str) -> dict:
        # call your API here
        return {"city": city, "forecast": "sunny"}

Then attach it to any device:

device = MyDevice.from_config("config.json")
device.use(WeatherPlugin())
device.run()

ROS2Plugin

ROS2Plugin makes any WISP device a natural-language interface for a ROS2 robot. It scans the live ROS2 graph at boot and registers capabilities for topics it recognises.

Requirements

  • ROS2 (Humble or later) installed on the same machine
  • rclpy importable (comes with ROS2 install)
  • Your robot's nodes running before WISP starts

Usage

from wisp import WispDevice
from wisp.plugins.ros2 import ROS2Plugin
import rclpy

rclpy.init()

class TurtleBot(WispDevice):
    device_name = "turtlebot3"
    device_description = "TurtleBot3 Burger mobile robot."

device = TurtleBot.from_config("config.json")
device.use(ROS2Plugin())
device.run()
Teleop example: Send "move forward" via Telegram — WISP routes it to move_forward capability → publishes a geometry_msgs/Twist on /cmd_vel.

Auto-discovered capabilities

The plugin maps topic patterns to capability names:

Topic patternCapabilityAction
/cmd_velmove_forward, move_backward, turn_left, turn_right, stopPublishes Twist
/navigate_to_posenavigate_toSends Nav2 goal
/scanread_lidarReads one scan
/battery_statebattery_levelReads battery
/camera/image_rawcapture_imageGrabs one frame
/imuread_imuReads IMU data
/odomget_positionReads odometry

The plugin only registers capabilities for topics that are currently live on the graph. If a topic isn't published, its capability is skipped.

Custom topic map

Override TOPIC_MAP in a subclass to add your robot's topics:

from wisp.plugins.ros2 import ROS2Plugin

class MyRobotPlugin(ROS2Plugin):
    TOPIC_MAP = ROS2Plugin.TOPIC_MAP + [
        {
            "topic": "/gripper/command",
            "capability": "open_gripper",
            "description": "Open or close the gripper. state: open|close.",
            "msg_type": "std_msgs/Bool",
        }
    ]

Writing your own plugin

Any Python capability — HTTP API, database, GPIO expander, CAN bus — can become a WISP plugin in minutes:

1
Subclass WispPlugin
from wisp import WispPlugin, CapabilitySpec

class HomeAssistantPlugin(WispPlugin):
    def __init__(self, base_url: str, token: str):
        self.base_url = base_url
        self.token = token
2
Implement attach()
    def attach(self, device) -> None:
        device.registry.register(CapabilitySpec(
            name="toggle_light",
            description="Toggle a Home Assistant light. entity: light.name.",
            params={"entity": {"type": "string"}},
            method=self._toggle,
        ))
3
Implement the method
    def _toggle(self, entity: str) -> dict:
        import urllib.request, json
        url = f"{self.base_url}/api/services/light/toggle"
        data = json.dumps({"entity_id": entity}).encode()
        req = urllib.request.Request(url, data=data,
              headers={"Authorization": f"Bearer {self.token}"})
        urllib.request.urlopen(req)
        return {"toggled": entity}
4
Attach and run
device.use(HomeAssistantPlugin(
    base_url="http://homeassistant.local:8123",
    token="your_long_lived_token",
))
device.run()
