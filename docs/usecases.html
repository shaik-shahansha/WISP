Common implementation patterns

Most use cases map cleanly to one of three patterns:

Pattern 1 — Sense & Report: Sensor readings wrapped in @capability. Users ask, WISP reads the hardware and replies. Zero side effects.
Pattern 2 — Sense & Act: Sensor read + conditional actuator control in one capability. "If it's above 30°C, turn on the fan." The AI chains capabilities or you implement the logic explicitly.
Pattern 3 — Plugin Integration: A third-party API or protocol (Modbus, MQTT, REST) wrapped in a WispPlugin. Attach it to any device without modifying the device class. 
# Pattern 3 — Wrapping a third-party system as a plugin
class ModbusPlugin(WispPlugin):
    def attach(self, device) -> None:
        device.registry.register(CapabilitySpec(
            name="read_power_draw",
            description="Read current power consumption in watts.",
            method=self._read,
        ))

    def _read(self) -> dict:
        # read register 0x0001 from Modbus device 1
        watts = self.client.read_holding_registers(1, 1).registers[0]
        return {"power_draw_w": watts}
