AI Providers

WISP works with Groq and OpenRouter out of the box. Both offer free tiers — no credit card needed to get started.

Recommendation: Start with Groq. It's the fastest and simplest to set up. Switch to OpenRouter if you want to experiment with other models.

Groq

Groq delivers inference on custom LPU hardware. The free tier is extremely fast — most WISP commands complete in under 500 ms end-to-end.

Setup

1
Create a free account

Go to console.groq.com and sign up. No credit card required.

2
Create an API key

Click API Keys → Create API Key. Copy the key starting with gsk_.

3
Add to config.json
{
  "ai": {
    "provider": "groq",
    "api_key": "gsk_your_key_here",
    "model": "llama-3.3-70b-versatile"
  }
}

OpenRouter

OpenRouter aggregates hundreds of AI models behind a single OpenAI-compatible API. Use it to access models not available on Groq.

Setup

{
  "ai": {
    "provider": "openrouter",
    "api_key": "sk-or-v1-your-key",
    "model": "meta-llama/llama-3.3-70b-instruct:free"
  }
}

Note: Free models on OpenRouter may have rate limits or context limits. If WISP fails with a 429 or 400 error, try a different model.

Custom AI provider

Implement the AIProvider interface to integrate any LLM:

from wisp.ai.providers import AIProvider

class OllamaProvider(AIProvider):
    def __init__(self, model: str = "llama3.2"):
        self.model = model

    def complete(self, system: str, user: str) -> str:
        import urllib.request, json
        payload = {
            "model": self.model,
            "messages": [
                {"role": "system", "content": system},
                {"role": "user", "content": user},
            ],
            "stream": False,
        }
        req = urllib.request.Request(
            "http://localhost:11434/api/chat",
            data=json.dumps(payload).encode(),
            headers={"Content-Type": "application/json"},
        )
        resp = json.loads(urllib.request.urlopen(req).read())
        return resp["message"]["content"]

Then pass it to the AI client when constructing your device — or open a GitHub issue to propose it as a built-in provider.

How grounding works

Every prompt sent to the AI contains a live, exact list of capabilities the device has right now. The AI cannot invent capabilities that aren't in the list.

System prompt sent to the model:

You are the AI controller for a device called "smart_home".
The device can only perform the following actions (JSON list):

[
  {"action": "read_temperature", "description": "Read temperature and humidity.", "params": {}},
  {"action": "set_fan",          "description": "Turn fan on or off.",            "params": {"state": "string"}},
  {"action": "set_led",          "description": "Set LED brightness 0-100.",       "params": {"brightness": "int"}}
]

Respond ONLY with a JSON object:
{"action": "<action_name>", "params": {}}

If the user's request cannot be fulfilled, respond:
{"action": "error", "message": "No capability for: <request>"}

This grounding is the core anti-hallucination mechanism. No fine-tuning, no prompt engineering required on your part.
